Etiquetas adicionales

En HTML5 se han definido o redefinido algunas etiquetas interesantes, que pueden ser muy útiles para representar información de manera que sea accesible para personas discapacitadas y sistemas.

TIME

La etiqueta time permite especificar tanto fechas, como horas e instantes en el tiempo. Algunos ejemplos de uso de dicha etiqueta son: Un mes concreto:2015-11 Una fecha: 2015-11-12 Una fecha en un año cualquiera: 11-12 Una hora: 14:54:39 Un instante en el tiempo: 2015-11-12T14:54:39 La definición de un offset de zona:-0800 Un instante en el tiempo global (con su offset):2015-11-12T06:54:39.929-0800 Una semana concreata de un año: 2015-W46 Una duración:4h 18m 3s

Etiquetas sobre el progreso de un proceso o accción

El progress se usa para describir el estado actual de un proceso de cambio que se completará.

El elemento del meter representa un elemento cuyo rango es conocido, lo que significa que tiene valores mínimo y máximo definidos, con rangos de valores asociados a niveles bajos y altos. Ejemplos: uso del disco duro, o una fracción de una población con derecho a voto; ambos tienen un valor máximo definido

Ejemplos:

Su tarea está en prograso

Estado: 10%

Uso total de disco: 3 GB

Etiqutas para glosarios y definiciones

El elemento dl (lista de definiciones), junto con sus hijos asociados dt (term) y dd (descripción), se ha redefinido en la especificación HTML5. En HTML5, estas listas ya no se llaman "listas de definición"; ahora son las "listas de descripción" o "listas de asociación" que suenan más genéricas. Deben usarse para marcar cualquier tipo de pares nombre-valor, incluidos términos y definiciones, temas y valores de metadatos, y preguntas y respuestas.
Selector:
El elemento (s) objetivo.
Propiedad:
La característica utilizada para agregar estilo al elemento objetivo, definido antes de dos puntos.
Valor:
El valor dado a la propiedad especificada, declarada después del simbolo de dos puntos.

Accesibilidad. WCAG y WAI-ARIA

Eesde 2011, el W3C ha hecho grandes progresos para facilitar a los autores la escritura de páginas web más accesibles.

WCAG

WCAG existe para proporcionar: "Un único estándar compartido para el acceso a contenido web que satisfaga las necesidades de individuos, organizaciones y gobiernos a nivel internacional".

Cuando se trata de crear páginas web normales (a diferencia de las aplicaciones web de una sola página y similares) tiene sentido concentrarse en las directrices de WCAG. Ofrecen una serie de pautas (en su mayoría de sentido común) sobre cómo garantizar que se pueda acceder a su contenido web. Cada recomendación se clasifica como un nivel de conformidad: A, AA o AAA. Para obtener más información sobre estos niveles de conformidad, consulte http://www.w3.org/TR/UNDERSTANDING-WCAG20/conformance.html#uc-levels-head.

Probablemente descubrirá que ya está siguiendo a muchas de las directrices, como proporcionar texto alternativo para imágenes, por ejemplo. Sin embargo, puede obtener un breve resumen de las pautas en http://www.w3.org/WAI/WCAG20/glance/Overview.html y luego crear su propia lista de verificaciones de referencia rápida personalizada en http://www.w3.org/WAI/WCAG20/quickref/.

Yo os Animaría a todos a pasar una o dos horas mirando la lista. Muchas de las pautas son simples de implementar y ofrecen beneficios reales a los usuarios

WAI-ARIA

El objetivo de WAI-ARIA es principalmente resolver el problema de hacer que el contenido dinámico en una página web sea accesible. Proporciona un medio para describir roles, estados y propiedades para widgets personalizados (secciones dinámicas en aplicaciones web) para que sean reconocibles y utilizables por los usuarios de tecnología de asistencia.

Por ejemplo, si un widget en pantalla muestra el precio de las acciones de una compañia que se actualiza constantemente, ¿cómo lo sabría un usuario ciego al acceder a la página? WAI-ARIA intenta resolver estos mismos problemas.

Lo más fácil que puede hacer para ayudar a las tecnologías de asistencia es utilizar los elementos correctos cuando sea posible. Un elemento de encabezado va a ser mucho más útil que div class = "Header". De manera similar, si tiene un botón en su página, use el elemento button (en lugar de un enlace u otro elemento diseñado para que parezca un botón).

ARIA define una serie de roles de referencias que ayudan a los sistemas de tecnología asisitiva a simular la experiencia para sus usuarios. Se encuentra disponible una lista completa de los roles y una descripción sucinta de su idoneidad de uso en http://www.w3.org/TR/wai-aria/roles.
